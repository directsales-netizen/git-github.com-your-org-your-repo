## 🎯 What This Is

The Premium TechNoir Development Bible is a comprehensive, professional documentation system that enables consistent, high-quality development over time. It contains:

- **Company Foundation** – Mission, vision, values, brand identity
- **Master Instructions** – Coding standards, quality requirements, working agreements
- **Product Requirements** – Detailed PRDs for every feature
- **Prompt Library** – Structured prompts for each development phase
- **Design System** – Colors, typography, components, icons
- **AI Manual** – Assistant personality and capabilities
- **Operational Guides** – Inventory, marketing, deployment

This Bible ensures that every piece of work—whether written by a human developer or Claude—reflects Premium TechNoir's values and professional standards.

## 🎯 Key Principles

### 1. **Context is Everything**
Every prompt includes full business context so Claude doesn't have to guess.

### 2. **Standards are Non-Negotiable**
Coding standards, quality requirements, accessibility—all defined upfront.

### 3. **Acceptance Criteria Are Verifiable**
Every task has a checklist. Don't mark it done until all criteria are met.

### 4. **Design System is Source of Truth**
No hardcoded colors, no one-off spacing values. All design decisions reference the design system.

### 5. **Sustainability in Code**
Write code that's maintainable, scalable, well-documented. That's what sustainability means here.

### 6. **Security First**
Assume every input is malicious. Validate everything. Never commit secrets.

### 7. **Accessibility is Built-In**
WCAG 2.1 AA minimum, always. Not an afterthought.

## 🤝 Working with Claude

When you want Claude to build something:

1. **Provide full context:**
   ```
   "Build [feature] following Premium TechNoir standards.
   
   Context:
   - Company is sustainability-focused, transparent, premium
   - See Volume 1 for brand values: [link]
   - See Volume 2 for coding standards: [link]
   - Use design system from DESIGN_SYSTEM.md
   
   Reference existing code:
   - Navigation component: [path]
   - Product card: [path]
   - Design utilities: [path]"
   ```

2. **Include acceptance criteria:**
   ```
   "This task is complete when:
   - [ ] Criterion 1
   - [ ] Criterion 2
   - [ ] Accessibility: WCAG 2.1 AA
   - [ ] Performance: Lighthouse > 90"
   ```

3. **Reference the Bible:**
   ```
   "Follow all standards in docs/VOLUME_2_CLAUDE_MASTER_INSTRUCTION.md
   Follow design system at DESIGN_SYSTEM.md
   Use prompt from docs/VOLUME_4_PROMPT_LIBRARY.md"
   ```

## ✅ Quality Checklist

Before considering any task "done":

- [ ] **Acceptance criteria:** All checkboxes marked
- [ ] **Code quality:** Follows Volume 2 standards
- [ ] **Testing:** Unit tests pass, E2E tests pass
- [ ] **Accessibility:** WCAG 2.1 AA verified
- [ ] **Performance:** Lighthouse > 90
- [ ] **Design system:** All colors, spacing, typography from system
- [ ] **Documentation:** Code commented, complex decisions documented
- [ ] **Security:** No secrets committed, input validation present
- [ ] **Responsive:** Tested at 375px, 768px, 1024px, 1440px

## 📞 FAQ

**Q: Do I have to follow all these standards?**
A: Yes. This is what makes the difference between "code that works" and "code that lasts."

**Q: Can I skip a phase?**
A: No. Each phase builds on the previous one. Skipping ahead creates gaps.

**Q: What if the business changes?**
A: Update Volume 1. The standards and prompts will still apply.

**Q: Can I use different technologies?**
A: No. This Bible is built around Next.js, TypeScript, Tailwind CSS, Prisma. These are non-negotiable.

**Q: How often do I check this Bible?**
A: Every time you start work. Always have the relevant volumes open.

**Q: What if I find a mistake?**
A: Fix it and update this Bible. It's a living document.

## 💡 Pro Tips

1. **Keep all volumes open while working** – Reference them constantly
2. **Use the prompt library as-is** – Don't paraphrase, the context matters
3. **Verify acceptance criteria before moving on** – Don't skip verification
4. **Build one prompt at a time** – Chain them, don't parallel process
5. **Reference previous work in new prompts** – Show Claude what's already been built
6. **Test everything** – Unit tests, E2E tests, manual testing, accessibility
7. **Commit often** – Small commits with clear messages
8. **Update the Bible** – If you discover something new, add it
